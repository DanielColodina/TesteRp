# EXPLICAÇÃO MUITO SIMPLES DO SISTEMA - PARA ALUNOS

Olá alunos! Vou explicar este sistema de computador de uma forma bem simples, como se fosse uma conversa. Vamos ver o que eu fiz e como funciona cada parte.

## O QUE É ESTE SISTEMA?

Imagine que você tem uma construtora. Este sistema é como um "caderninho mágico" no computador que ajuda a organizar:
- Os clientes (usuários)
- As obras que estão sendo feitas
- Quem está fazendo o quê
- Quanto dinheiro entra e sai
- E muito mais!

## COMO FUNCIONA O SISTEMA? (EXPLICAÇÃO BEM SIMPLES)

O sistema funciona como uma casa com vários cômodos:

1. **Porta de entrada (Login)**: Só quem tem a chave pode entrar
2. **Sala de estar (Dashboard)**: Mostra tudo que está acontecendo
3. **Cozinha (Usuários)**: Onde cuidamos dos clientes
4. **Quarto (Obras)**: Onde organizamos os trabalhos
5. **Banheiro (CONTROLEGERAL)**: Um quartinho extra para coisas especiais

Agora vamos ver o que EU fiz para melhorar este sistema.

## 🔧 O QUE EU CONSERTEI E MELHOREI

### 1. PRIMEIRO PROBLEMA: "ERRO AO DESVINCULAR OBRA"

**O problema era**: Quando tentavam tirar uma obra de um usuário, dava erro porque o banco de dados não deixava "esvaziar" o campo do usuário.

**O que eu fiz**: Mudei o banco de dados para permitir que o campo fique vazio.

**Código que eu executei no banco**:
```sql
USE rp_empreendimentos;
ALTER TABLE obras MODIFY COLUMN usuario_id INT NULL;
```

**Explicação simples**:
- `USE rp_empreendimentos;` = "Ei banco, vamos trabalhar na pasta 'rp_empreendimentos'"
- `ALTER TABLE obras` = "Vamos mudar a tabela 'obras'"
- `MODIFY COLUMN usuario_id INT NULL` = "O campo 'usuario_id' agora pode ficar vazio (NULL)"

---

### 2. SEGUNDO PROBLEMA: "ERRO AO EXCLUIR USUÁRIO"

**O problema era**: Quando excluíam um usuário, o sistema tentava registrar no "diário" (auditoria) DEPOIS de apagar o usuário, mas o banco reclamava porque o usuário já não existia mais.

**O que eu fiz**: Mudei a ordem - agora registra no diário ANTES de apagar.

**Código que eu mudei** (no arquivo `src/controllers/usuarioController.js`):

ANTES (errado):
```javascript
await User.deleteById(id);     // Apaga primeiro
// Registrar auditoria         // Depois tenta registrar (erro!)
await Auditoria.log(req.session.adminId, id, 'USUARIO_DELETADO', 'usuario', usuario.nome, null);
```

DEPOIS (certo):
```javascript
// Registrar auditoria         // Registra primeiro
await Auditoria.log(req.session.adminId, id, 'USUARIO_DELETADO', 'usuario', usuario.nome, null);
await User.deleteById(id);     // Depois apaga
```

**Explicação simples**:
- `await Auditoria.log(...)` = "Escreve no diário quem apagou quem e quando"
- `await User.deleteById(id)` = "Agora sim, apaga o usuário"

---

### 3. TERCEIRO PROBLEMA: "EMAIL JÁ CADASTRADO"

**O problema era**: O sistema deixava cadastrar dois usuários com o mesmo email, causando erro.

**O que eu fiz**: Adicionei uma checagem antes de cadastrar.

**Código que eu adicionei** (no arquivo `src/models/User.js`):

```javascript
// Verificar se email já existe
const [existing] = await db.execute('SELECT id FROM usuarios WHERE email = ?', [email]);
if (existing.length > 0) {
  throw new Error('Email já cadastrado');
}
```

**Explicação simples**:
- `SELECT id FROM usuarios WHERE email = ?` = "Procura se já existe alguém com este email"
- `if (existing.length > 0)` = "Se encontrou alguém..."
- `throw new Error('Email já cadastrado')` = "Para tudo e mostra mensagem de erro"

---

### 4. QUARTO PROBLEMA: CONTROLEGERAL NÃO FUNCIONAVA

**O problema era**: O módulo CONTROLEGERAL não conseguia falar com o servidor porque não estava "logado".

**O que eu fiz**: Ensinei o JavaScript a mandar o "cookie de login" junto.

**Código que eu adicionei** (no arquivo `CONTROLEGERAL/frontend/script.js`):

```javascript
// Função auxiliar para fetch com credenciais
async function apiFetch(url, options = {}) {
    return fetch(url, {
        ...options,
        credentials: 'include',  // ← Esta linha é importante!
        headers: {
            'Content-Type': 'application/json',
            ...options.headers
        }
    });
}
```

**Explicação simples**:
- `credentials: 'include'` = "Manda junto o cookie de login"
- Agora quando o CONTROLEGERAL pede dados, o servidor sabe quem é

---

### 5. QUINTO PROBLEMA: CONTROLEGERAL FEIO

**O problema era**: O CONTROLEGERAL tinha aparência diferente do resto do sistema.

**O que eu fiz**: Fiz ele usar o mesmo "figurino" (CSS) do sistema principal.

**Código que eu adicionei** (no arquivo `CONTROLEGERAL/frontend/index.html`):

```html
<link rel="stylesheet" href="/css/style.css">  ← CSS do sistema principal
<link rel="stylesheet" href="style.css">       ← CSS específico do CONTROLEGERAL
```

**E também mudei todo o CSS** (no arquivo `CONTROLEGERAL/frontend/style.css`):

```css
body {
    background: linear-gradient(135deg, #0f172a 0%, #1a1f35 100%);  /* Fundo escuro igual */
    color: #e2e8f0;  /* Texto claro */
    font-family: 'Segoe UI', Tahoma, Geneva, Verdana, sans-serif;  /* Mesma fonte */
    margin: 0;
    padding: 20px;
    min-height: 100vh;
}
```

**Explicação simples**:
- `linear-gradient(135deg, #0f172a 0%, #1a1f35 100%)` = "Fundo com cores escuras e azuis"
- `color: #e2e8f0` = "Texto quase branco"
- Agora parece o mesmo sistema!

## 📁 ESTRUTURA DOS ARQUIVOS (EXPLICADA BEM SIMPLES)

Vamos ver como os arquivos estão organizados, como se fosse uma casa:

### `src/` = A "casa principal" do código
- `controllers/` = "Os chefs" - decidem o que fazer com os pedidos
- `models/` = "A despensa" - guardam e organizam os dados
- `routes/` = "O mapa" - mostram os caminhos para chegar aos lugares
- `views/` = "As janelas" - mostram as páginas bonitas
- `middlewares/` = "Os porteiros" - verificam se você pode entrar

### `public/` = Arquivos que todo mundo pode ver
- `css/style.css` = "O figurino" - deixa tudo bonito
- `js/` = "Os ajudantes" - fazem as páginas funcionarem

### `CONTROLEGERAL/` = Um "anexo" da casa
- `frontend/` = A parte da frente (páginas web)
- `backend/` = A parte de trás (servidor)

### `scripts/` = "Caixa de ferramentas" para manutenção

## 🚀 COMO O SISTEMA FUNCIONA NO DIA A DIA

### Quando alguém entra no sistema:

1. **Login**: Digita email e senha
2. **Verificação**: O sistema confere se está certo
3. **Sessão**: Cria um "crachá invisível" no navegador
4. **Dashboard**: Mostra a tela principal com estatísticas

### Quando cadastra um usuário:

1. **Formulário**: Preenche nome, email, etc.
2. **Validação**: Confere se email já existe
3. **Banco**: Salva no MySQL
4. **Auditoria**: Registra quem fez isso

### Quando usa o CONTROLEGERAL:

1. **Clica na aba**: Abre o módulo especial
2. **Carrega dados**: Pede informações do SQLite
3. **Mostra**: Exibe obras, materiais, etc.
4. **Salva**: Quando adiciona algo novo
